Section 01

Platform Overview

The Business Agents Platform is an AI agent deployment and management system for mid-market industrial manufacturers and specialty businesses. It provides a library of autonomous agents that monitor operations, generate recommendations, and execute approved actions — all with human-in-the-loop oversight. The platform includes a process intelligence database (760 processes, 13 industries, 40 agent blueprints), persona-specific dashboards, client engagement portals, and a Flask server that orchestrates agent runs.

Key Schema Notes

- industry_processes.relevance (not relevance_score)
- industries.code (not slug)
- agent_blueprints.authority_level stored lowercase: low / medium / high
- process_agent_actions.action_type values: monitor, analyze, generate, execute, coordinate
- process_agent_actions.trigger_type values: scheduled, event, threshold, manual
- process_agent_actions.integration values: read, write, read_write
- agent_blueprints.lifecycle_stage values: blueprint / scaffolded / sandbox / validated / pilot_ready / production
- Views available: v_process_full, v_opportunity_matrix, v_capability_matrix, v_blueprint_summary, v_feature_usage, v_run_history

Agent Architecture

Every agent follows the same two-tier pattern: a generic platform version in pilot-agents/, and an optional client-specific variant in pilot-agents/kimre/. The Flask server maps blueprint slugs to file paths via the AGENT_PATHS registry.

Agent Output Schema

Every agent — generic or client-specific — returns this structure on stdout. The server parses it to extract recommendations.

{ "agent": "replenishment-agent", // blueprint slug "run_at": "2026-03-14T21:00:00Z", // ISO 8601 "dry_run": true, // always true for sandbox "summary": { "items_processed": 24, "recommendations_generated": 6 }, "recommendations": [ { "rec_type": "reorder", // reorder | quote_draft | compliance_flag | alert | ... Section 06

Data Flow: Agent Run to HITL Outcome

The human-in-the-loop (HITL) cycle is the core interaction model. A dashboard triggers an agent run; the server captures recommendations; a human decides; the decision is written to the audit log.

sequenceDiagram participant D as Dashboard (Browser) participant S as Flask Server :8500 participant A as Agent (Python subprocess) participant DB as SQLite DB D->>S: POST /api/run/{slug} Note over S: Look up agent file path\nfrom AGENT_PATHS registry S->>A: subprocess.run(agent.py --dry-run) A-->>S: stdout: JSON output\n{recommendations: [...]} S->>DB: INSERT INTO pilot_runs\n(slug, run_at, output_json) S->>DB: INSERT INTO recommendations\n(run_id, rec_type, urgency, decision='pending') S-->>D: {outcome: 'success', recommendations: [...]} Note over D: Render HITL cards\napprove / reject per item D->>S: POST /api/recommendations/{id}/decide\n{decision: 'approved'} S->>DB: UPDATE recommendations\nSET decision='approved', decided_at=NOW() S->>DB: INSERT INTO approvals\n(recommendation_id, decision, decided_at) S-->>D: {decided: true, decision: 'approved'} Note over D: Show toast confirmation\nUpdate card state

Key Implementation Details

- All agents currently run with --dry-run flag; no production system writes occur
- _extract_recommendations() in server.py parses the agent's stdout JSON and normalizes rec_type/urgency fields
- The approvals table provides the full audit trail; every decision is timestamped and immutable
- If an agent subprocess fails (exit code != 0), the server returns a 500 with the stderr content and logs the failed run
- The orchestrator (orchestrator.py) can run multiple agents in sequence and merge their outputs into one combined JSON response

Section 08

Running the Platform

All services are independent. Start in whichever order suits your task. The Flask server is the primary entry point for the full demo experience.

Step 1 — Start Mock Integration APIs

Provides fake SAP S/4HANA and MES endpoints that the pilot agents read from during dry runs.

# From workspace root
cd "/Users/keith_ai/Documents/Agentic Projects/sandbox-systems"
./start-all.sh

# SAP mock: http://localhost:3001
# MES mock: http://localhost:3002

Step 2 — Start the Flask Server

Serves all dashboards, client portals, and API routes. This is the main app entry point.

# Separate terminal
cd "/Users/keith_ai/Documents/Agentic Projects/dashboards"
python3 server.py

# Platform: http://localhost:8500
# API health: http://localhost:8500/api/health

Step 3 — Run a Pilot Agent (CLI)

Agents can also be run directly from the command line for debugging or development.

# Single agent, dry run
cd "/Users/keith_ai/Documents/Agentic Projects/pilot-agents"
python3 replenishment_agent.py --dry-run

# Multi-agent via orchestrator
python3 orchestrator.py --agents replenishment_agent,demand_forecast_agent --dry-run

# Kimre-specific agent
python3 kimre/rfq_quote_agent.py --dry-run
